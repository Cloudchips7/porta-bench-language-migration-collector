# 语言变化迁移样本标注规范 v1

## 1. 目标

这份规范只服务于 Porta-Bench 场景二：`语言变化带来的迁移`。  
第一阶段优先稳定 `py2 -> py3`，第二阶段再扩展到 `cpp -> python`。

## 2. taxonomy

| subtype | migration_type | 含义 |
| --- | --- | --- |
| `py2_py3` | `version_migration` | 同一种语言的版本迁移 |
| `cpp_python` | `cross_language_migration` | 跨语言实现迁移 |

不要把这两类混在同一统计口径里。

## 3. 必要纳入条件

1. PR 的根因必须直接来自语言或语言版本变化。
2. 必须能标出 source/target language 或 version。
3. 改动主体应触及源码、运行时语义、构建适配或测试适配。
4. 样本后续应有机会改写成 issue，并具备测试验证空间。

## 4. 全局排除条件

1. 纯文档 PR。
2. 纯 CI PR。
3. 纯 dependency bump PR。
4. 机器人自动生成的 PR。
5. 与语言迁移无关的普通 bugfix / refactor。

## 5. py2 -> py3 判定规则

### 正例

1. PR 明确提到 Python 2 到 Python 3 兼容或迁移。
2. 改动涉及 `xrange`、`iteritems`、`unicode`、`bytes`、`raw_input`、`basestring`、`__future__`、`2to3`、`futurize` 等典型迁移点。
3. 测试或构建改动是配合真实迁移，而不是唯一改动。

### 负例

1. 只是支持 Python 3.11/3.12/3.13/3.14 的版本升级。
2. 只是 dependency bump。
3. 只是 CI matrix 改动。
4. 只是文档或 metadata 调整。

## 6. cpp -> python 判定规则

### 正例

1. PR 明确把已有 C/C++ 实现迁移、替换或重写为 Python 实现。
2. 修改前后仍然是同一功能模块或工具。
3. 文件层面能看到 `.c/.cpp/.h` 与 `.py` 的迁移关联。

### 负例

1. 只是 binding、wrapper、pybind11、ctypes、cffi、SWIG、Cython。
2. 只是改构建脚本。
3. 只是删除旧 C/C++ 文件，却没有对应 Python 实现。
4. 大规模重写到已经不适合作为单个 benchmark task。

## 7. CSV 字段怎么填

| 字段 | 填写建议 |
| --- | --- |
| `manual_label` | `positive` / `negative` / `uncertain` |
| `migration_type` | `version_migration` 或 `cross_language_migration` |
| `source_language` / `target_language` | 如 `python`、`c++` |
| `source_version` / `target_version` | 如 `2`、`3`、`3.12` |
| `reproducible` | `yes` / `no` / `unknown` |
| `issue_rewrite_ready` | `yes` / `needs_check` / `no` |
| `leakage_risk` | `low` / `medium` / `high` |
| `test_framework` | `pytest` / `unittest` / 空 |
| `build_system` | `setuptools` / `poetry` / `cmake` / 空 |

## 8. 标注顺序

1. 先看标题和 PR body，判断根因是否真是语言迁移。
2. 再看 changed files，确认是否真的触及代码和测试。
3. 再填 source/target 与 migration_type。
4. 最后评估是否可复现、是否适合改写成 issue、是否存在泄露风险。

## 9. 一致性要求

1. 两人先对同一批 20 条样本做重叠标注。
2. 如果一致率低于 85%，先修订规范，再继续扩大样本。
3. `uncertain` 样本不直接丢弃，进入二次复核。
