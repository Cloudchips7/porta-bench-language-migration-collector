语言迁移样本标注演示页

这页用一个真实样本演示 review_zh.csv 里需要人工填写的标签应该怎么标。 左边是样本证据，右边是需要你们点选或填写的字段。这个例子故意选了一个比较“标准”的正例， 这样你可以先建立直觉，再回头看那些更模糊的样本。

样本信息

样本 IDcpp_python__arendst__Tasmota__pr24599
子类型C/C++ 到 Python
仓库arendst/Tasmota
Stars24254
PR 标题Berry solidification made possible with Python runtime
PR 链接#24599
为什么这个样本适合做演示：
PR 正文明确写了 The Berry runtime was ported from C to Python， 而且新增了大量 Python 文件，语义上非常接近“把同一块核心逻辑从 C/C++ 翻到 Python”。
PR 正文里的关键信号
  • 明确说了：Berry runtime was ported from C to Python。
  • 说明迁移目标不是随便加个脚本，而是把 Berry 运行时的固化阶段搬进 Python 工具链。
  • 还说到 byte-to-byte compatible，说明新旧逻辑在功能上高度对应。
关键改动文件
  • lib/libesp32/berry/berry_port/__init__.py
  • lib/libesp32/berry/berry_port/__main__.py
  • lib/libesp32/berry/berry_port/be_filelib.py
  • 同一个 PR 同时保留了原 Berry 的 C 实现上下文，并新增成体系的 Python 端口代码。
这个例子怎么判断“不是 wrapper/binding 噪声”：
如果只是 wrapper，通常会看到 pybind、bindings、JNI 一类词， 或者只是把 C API 包一层给 Python 调。这个例子不是，它直接说“runtime 从 C 迁到 Python”， 而且新增的是完整 Python 运行时入口和模块。
本例推荐标注思路
  1. manual_label：标成 positive。因为迁移意图明确，且改动触及核心实现。
  2. implementation_scope：标成 partial_feature_migration。因为迁移的是 Berry runtime/solidification 这一块，不是整个 Tasmota 仓库全部从 C/C++ 变成 Python。
  3. logic_equivalence_scope：标成 same_logic_translation。PR 自己说明新旧实现是 byte-to-byte compatible，说明逻辑对应关系很强。
  4. source_language / target_language：分别是 c++ 和 python。这里项目统一把 C 和 C++ 归到 c++ 这一类。
  5. migration_pattern：可以写成“Berry 运行时从 C 迁移到 Python，并把固化流程转入 Python 工具链”。
  6. reproducible：保守写 unknown。因为虽然 PR 说 tested，但我们还没有亲自复现实验。
  7. issue_rewrite_ready：先写 needs_check。因为还要确认能不能单独抽成一个清晰任务。
  8. leakage_risk：建议 medium。PR 明说了迁移方向，但没有把完整 patch 解法直接暴露成教程。
  9. exclude_reason：正例留空。
  10. cross_check_status：第一次标注先填 pending。
  11. notes：写清“为什么是正例、为什么是 partial 而不是 full”。

交互式标注区

右侧这些就是你们真正需要人工填写的重点字段。选完之后，下面会自动生成一份建议结果，方便你复制回 CSV。

1. manual_label 人工标签

决定这条样本最后收不收。正例是“真迁移”，负例是“噪声”，不确定是“需要二次复核”。

2. implementation_scope 实现范围

判断迁移的是一个功能模块，还是整个仓库的主体实现。

3. logic_equivalence_scope 逻辑对应关系

判断新旧实现是不是同一逻辑的翻译，还是只替换了一部分，或者根本看不清。

4. source / target language

跨语言样本通常已经有默认值，但人工需要确认方向有没有搞反。

5. migration_pattern 迁移模式说明

用一句人话写清楚“从什么迁到什么、迁的是哪块逻辑”。这个字段很适合后面做汇报和抽样分析。

6. reproducible / issue_rewrite_ready / leakage_risk

这三个字段决定后面能不能把它做成 benchmark 任务。

7. reviewer / cross_check_status / notes

这是人工流程字段。谁标的、是否复核过、为什么这么标，都放这里。

这个例子的推荐答案 manual_label = positive
implementation_scope = partial_feature_migration
logic_equivalence_scope = same_logic_translation
什么时候改成 negative 如果你发现它只是 wrapper、binding、SDK、bridge，或者根本没有“同一逻辑替换”，就应该标成 negative。
