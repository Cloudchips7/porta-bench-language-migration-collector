# 流程修订说明

## 1. 当前流程里原本合理的地方

| 模块 | 判断 | 说明 |
| --- | --- | --- |
| 单脚本多阶段 | 合理 | 对当前课程项目规模来说，`collect -> enrich -> export-review -> apply-review -> package` 已经足够清晰。 |
| 场景范围 | 基本合理 | 已经把任务收敛到 `py2_py3` 和 `cpp_python`，没有把 OS、框架迁移混进来。 |
| 条件过滤 | 合理 | stars、license、repo size、commit 数这些基础条件过滤是必要的。 |
| review CSV + processed jsonl | 合理 | 这符合 benchmark 数据工程的基本形态。 |

## 2. 当前流程里不够合理的地方

| 模块 | 问题 | 本次修订 |
| --- | --- | --- |
| taxonomy | `py2_py3` 和 `cpp_python` 语义差异很大，但之前没有单独标记 `migration_type`。 | 增加 `migration_type`，区分 `version_migration` 和 `cross_language_migration`。 |
| 标注字段 | review CSV 只够“筛 PR”，不够“做 benchmark”。 | 增加 `test_framework`、`build_system`、`reproducible`、`issue_rewrite_ready`、`leakage_risk`、`notes`。 |
| 标注规范 | 之前没有一份正式的样本标注规范。 | 新增 `语言变化迁移样本标注规范_v1.md`。 |
| 自动过滤 | 之前对 bot PR、纯 Python 3.x 版本升级、wrapper/binding 噪声处理不够。 | 增加更严格的噪声过滤规则。 |
| enrich 失败可见性 | 之前 enrich 失败只打日志，不进入结构化中间产物。 | enrich 失败现在会写入 `enriched_index.jsonl` 和进度看板。 |
| 可视化进度 | 之前需要手工翻 jsonl/log 才知道流程卡在哪。 | 每次阶段结束自动更新 `项目进度看板.md` 和 subtype 看板。 |

## 3. 这次新增的关键产物

1. `语言变化迁移样本标注规范_v1.md`
2. `项目进度看板.md`
3. `data/stats/py2_py3_progress.md`
4. `data/stats/cpp_python_progress.md`
5. 更新后的 `configs/review_schema.json`

## 4. 修订后的真实流程

```text
collect
  ↓
更严格的自动过滤
  - bot PR
  - py3-only support noise
  - binding/wrapper noise
  ↓
enrich
  - 保存 metadata
  - 记录 r0/rn 快照
  - 如果 clone 失败，结构化记录错误
  ↓
export-review
  - 导出 benchmark 导向字段
  ↓
apply-review
  - 根据人工标签沉淀 processed candidates
  ↓
package
  - 生成 stats
  - 生成项目看板
```

## 5. 当前最真实的瓶颈

不是 query，也不是 review schema。  
当前最主要的真实瓶颈是：

1. GitHub Search API 可用。
2. GitHub PR 元数据 API 可用。
3. `git clone https://github.com/...` 在当前网络环境下经常超时。

这意味着：

- `collect` 是通的。
- benchmark 结构也基本通了。
- 现在真正限制样本继续往下走的是仓库快照下载。

## 6. 下一步最值得做的事

1. 给 `enrich` 增加 snapshot fallback，不只依赖 `git clone`。
2. 先把 `py2_py3` 的 query 再做一轮精修，尽量把候选质量抬高。
3. 一旦快照网络问题缓解，立刻开始人工标注，不要等两个子场景都“完美”再动。
